## 2. 문제 정의와 현재 근거

### 2.1 확정된 사용자 관찰

HWP 재현 문서의 첫 페이지에서 `1.1.1` 문단 끝에 `1`을 반복 입력하면, 기존 줄 경계를
지난 뒤 캐럿과 새 문자가 즉시 보이지 않거나 잠시 뒤 사라진 것처럼 보인다. 같은 위치에서
Enter를 입력하면 페이지가 다시 그려지면서 앞서 입력한 문자가 나타난다.

Enter 후 내용이 보이는 현상은 실제 모델 데이터 손실보다 파생 레이아웃 또는 Canvas 표시
상태의 불일치를 시사하지만, 시간축의 모델 text와 저장 결과를 확인하기 전에는 데이터 보존을
확정하지 않는다. HWPX는 동일 원인 여부를 확인할 필수 검증 대상이며, 현재 사용자 관찰이
HWPX에서도 확정됐다고 전제하지 않는다.

재현 대상의 native 경로는 다음과 같다.

```text
section=0 / parent paragraph=0 / table control=2 / cell=2
cell paragraph=5 / initial char offset=130
```

### 2.2 코드에서 확인된 상태 전이

```text
작은 같은 셀 입력
→ InsertTextCommand
→ insertTextInCellDeferredPagination
→ 모델 text·LINE_SEG·후속 셀 문단 vpos 갱신
→ section dirty + page-tree cache 무효화, pagination은 보류
→ 현재 cursor page 한 장만 document-page-invalidated(text-edit)
→ rAF 단위 Canvas 갱신 + 조건부 static layer 재사용
→ 조건 충족 시 800ms 뒤 static layer 검증 렌더
```

현재 page-local 판정은 같은 셀·같은 before/after page, 8자 이하, 줄바꿈·탭 없음만
확인한다. 입력 전후 줄 수, 세로 범위, pagination fragment 또는 실제 영향 페이지 범위는
확인하지 않는다.

Rust deferred 입력은 새 모델과 새 `LINE_SEG`를 만든 뒤 section과 page-tree cache를
무효화하지만 즉시 pagination하지 않는다. 따라서 새 문단 줄 구조와 이전 pagination
fragment가 공존할 수 있다. Studio는 cursor rect가 가리키는 한 페이지만 다시 그리며,
115쪽 문서는 30쪽 자동 flush 제한을 넘으므로 저장·인쇄·구조 편집 전까지 deferred
pagination이 계속 pending일 수 있다.

800ms static-layer 검증은 항상 실행되는 것이 아니라 `flow-static`, BehindText 또는
InFrontOfText 재사용이 실제 발생한 경우에만 예약된다. 따라서 타이머가 증상의 원인인지,
이미 stale인 상태를 다시 드러내는 경로인지는 계측 전까지 가설로 둔다.

Enter는 문단 분할, pending pagination flush, 즉시 pagination 및 full
`document-changed`를 함께 일으킬 수 있다. 따라서 “Enter 후 복구”만으로 pagination과
Canvas refresh 중 어느 쪽이 원인인지 판정하지 않는다.

### 2.3 사실과 가설의 경계

| 분류 | 내용 |
|------|------|
| 확정 | deferred 입력 뒤 새 모델·`LINE_SEG`와 이전 pagination fragment가 공존할 수 있다. |
| 확정 | page-local 경로는 현재 cursor page 한 장만 무효화하고, 115쪽 문서는 자동 flush하지 않는다. |
| 1순위 가설 | 줄 수 또는 fragment 경계가 바뀌는 입력에서 단일 페이지 invalidation이 부족하다. |
| 2순위 가설 | stale pagination에서 얻은 cursor page와 `cellOverflowed` 판단이 fallback을 늦춘다. |
| 조건부 가설 | static layer 검증이나 지연 이미지 재렌더가 최신 Canvas를 오래된 상태로 덮는다. |
| 미확정 | 실제 모델 데이터가 손실되는지, HWPX에서도 동일한 표시 경로가 실패하는지 여부 |

## 3. 목표

1. 줄 수가 4줄에서 5줄로 바뀌는 경계를 포함한 연속 입력을 브라우저에서 결정적으로
   재현하고 시간축 관측 계약을 고정한다.
2. 모델, `LINE_SEG`, pagination fragment/page tree, cursor 및 합성 Canvas 중 최초로
   stale 상태가 발생하는 경계를 계측으로 확정한다.
3. 확정된 단일 경계를 가장 작은 범위에서 보완해, Enter나 수동 refresh 없이 새 입력과
   캐럿이 즉시 표시되도록 한다.
4. 매 입력마다 115쪽 전체 pagination을 수행하지 않으면서 정확성을 보장한다.
5. HWP/HWPX 저장·재로드와 #2185 한 글자 기준선, #1918 fast path 성능 안전성을 유지한다.

## 4. 수행 원칙

1. 데이터 보존 판정은 모델 text와 원본 형식 저장·재로드 결과를 최우선 권위로 삼는다.
2. 표시 판정은 page layer tree와 브라우저가 합성한 페이지 영역을 함께 사용한다. main
   flow canvas 하나만 캡처해 static/overlay layer를 누락하지 않는다.
3. 반복 입력 뒤 모델을 바꾸지 않는 `flushDeferredPagination()`과 full refresh 결과를
   정답 대조로 사용한다. Enter는 사용자 재현 증적으로만 사용한다.
4. stale 상태의 최초 경계를 확정하기 전 800ms 타이머 제거, cache 무효화 확대 또는
   pagination 정책 변경을 해결책으로 확정하지 않는다.
5. 수정이 전체 renderer/paginator 재설계로 확대되면 구현을 중단하고 범위 승인을 다시 받는다.

## 5. 수행 범위

### Stage 1. 결정적 재현과 시간축 상태 계측 고정

1. 최신 WASM과 Studio를 준비하고 HWP를 우선 재현한 뒤 HWPX에 같은 절차를 적용한다.
2. 고정 문서 경로와 offset 130에 캐럿을 놓고, 고정 입력 횟수 대신 실제
   `lineCount`가 4에서 5로 바뀔 때까지 `1`을 한 글자씩 입력한 뒤 2~3자를 더 입력한다.
3. 입력 전과 각 경계에서 다음 체크포인트를 기록한다.
   - 입력 동기 처리 직후
   - 2 rAF 뒤
   - 100ms 뒤
   - 850ms 뒤
   - 1.6초 뒤
   - 명시적 deferred pagination flush와 full refresh 뒤
4. 각 체크포인트에서 다음 상태를 하나의 레코드로 수집한다.
   - 셀 모델 text와 누적 suffix
   - cursor position, char offset, page, rect, `cellOverflowed`
   - 대상 문단 `LINE_SEG`의 줄 수·`text_start`·vpos·높이
   - deferred pagination pending 여부와 page count
   - 대상 문단이 나타나는 page fragment/page layer tree의 TextRun·좌표
   - invalidated page와 render context, filtered layer 호출
   - caret DOM과 합성된 페이지 영역 screenshot/pixel 비교
5. 입력 중 `flushDeferredPagination()` 호출 횟수를 기록해 page-local 경로가 유지되는지
   확인한다.

### Stage 2. 최초 불일치 경계와 영향 범위 확정

1. 다음 통제 대조를 분리해 어느 상태가 처음 stale해지는지 확인한다.
   - pagination 없이 full Canvas rebuild만 수행
   - 명시적 pagination flush 후 full refresh 수행
   - static layer 검증이 실제 예약된 경우 800ms 직전·직후 비교
2. 입력 전후 줄 수, 마지막 줄 세로 범위, 다음 셀 문단 vpos와 대상 TextRun이 포함된
   page 목록을 비교한다.
3. cursor page와 실제 영향 page 범위가 같은지 확인하고, page-local 판정이 줄/fragment
   영향도를 놓치는지 검증한다.
4. 원인별 후보를 통제 대조 결과로 좁힌다.
   - 편집 결과가 layout impact 또는 `requiresPagination` 신호를 반환하는 방식
   - before/after/affected page 범위만 무효화하는 방식
   - 제한된 경계에서만 보수적으로 pagination하는 방식
   - render/edit generation으로 stale timer와 cache 결과를 폐기하는 방식
5. 매 입력 전체 pagination 없이 정확성을 보장하는 최소안을 구현계획서에서 확정한다.

### Stage 3. 회귀 핀과 검증 계약 설계

1. 기존 `tests/issue_2185_korean_break_unit.rs`의 한 글자 기준선은 그대로 유지한다.
   - 한 글자 입력 후 줄 시작 `[0, 44, 84, 122]`
   - 다음 문단 `vpos = 17160`
   - 전체 115쪽과 저장·재로드 동일성
2. 별도 native #2214 회귀는 다음을 검증하도록 설계한다.
   - HWP/HWPX에서 4줄→5줄 전환까지 한 글자씩 deferred 입력
   - 매 입력 후 모델 text가 정확히 누적됨
   - 기존 네 줄 시작점 `[0, 44, 84, 122]`가 prefix로 보존됨
   - 반복 입력 후 줄 구조와 다음 문단 vpos가 명시적 flush 및 재로드 결과와 동일함
   - 새 줄이 생긴 뒤에는 다음 문단 `vpos=17160`을 고정하지 않음
   - 115쪽과 원본 형식 저장·재로드 결과 보존
3. 신규 Studio E2E는 실제 페이지 합성 화면을 기준으로 다음을 검증한다.
   - 입력마다 모델 text와 cursor offset 증가
   - 1~2 rAF 안에 layer tree, 표시 문자와 caret 위치 갱신
   - 850ms와 1.6초 뒤에도 새 입력이 되돌아가지 않음
   - explicit flush 전후 구조적 상태와 합성 페이지 crop이 동등함
   - 입력 중 전체 pagination flush 호출 0
4. page-local 판정 또는 Canvas fast path를 수정하면 관련 TypeScript 단위 계약을
   보강하되, 현재 source 정규식 테스트만으로 브라우저 E2E를 대체하지 않는다.
5. #1918의 정적 레이어 재사용과 빠른 연속 입력 성능이 퇴행하지 않는 대표 probe를
   함께 고정한다.

### Stage 4. 최소 수정, 전체 검증과 결과 분리

## 8. 완료 조건

- HWP/HWPX에서 줄 수가 4줄에서 5줄로 바뀌는 연속 입력이 결정적으로 재현되고 자동화된다.
- 모든 입력 문자가 1~2 rAF 안에 layer tree와 합성 화면에 나타나며 caret이 마지막 문자 뒤를
  가리킨다.
- 850ms와 1.6초 뒤에도 입력 문자, 줄 배치와 caret이 사라지거나 이전 상태로 되돌아가지 않는다.
- 모델 text, explicit flush, 원본 형식 저장·재로드 결과의 입력 문자열이 정확히 일치한다.
- explicit flush 전 page-local 결과와 flush 후 결과가 구조 및 합성 화면에서 동등하다.
- 기존 #2185 한 글자 `[0, 44, 84, 122]`, `vpos=17160`, 115쪽 기준선을 유지한다.
- 반복 입력 후에는 기존 네 줄 시작점을 prefix로 보존하고, 새 줄 및 다음 문단 vpos는 explicit
  flush와 재로드 결과에 일치한다.
- HWP/HWPX 모두 최종 115쪽을 유지한다.
- 입력마다 115쪽 전체 pagination을 수행하지 않는다.
- #1918 page-local fast path의 대표 입력 성능과 정적 레이어 표시가 퇴행하지 않는다.

`mydocs/working/task_m100_2214_stage{N}.md`를 같은 커밋에 두며, 작업지시자 승인 없이

다시 승인받는다. 이슈 close, push, 통합은 각각 별도 승인을 따른다.
